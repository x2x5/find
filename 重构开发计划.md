## 1. 项目目标

### 1.1 要解决的问题

1. **首屏加载慢** —— 现在打开页面要等 ~2MB 数据下载完才能用
2. **缓存失效** —— `?t=Date.now()` 让浏览器每次都重新下载所有数据
3. **HTML/CSS 难维护** —— `index.html` 3606 行、6 个内联 `<style>` 块、满天飞的 inline style
4. **没有类型检查** —— AI 写代码或自己改代码容易拼错变量名、字段名

### 1.2 期望达成

1. 首屏只下载需要的数据（默认"最近两年"约 300-500KB 而不是 2MB）
2. 缓存基于内容指纹（数据没改 → 浏览器直接走缓存）
3. 组件化的 React 代码，每个功能一个文件，易读易改
4. TypeScript 全覆盖，编辑器实时报错

### 1.3 不变的事情

1. 网站功能（搜索、筛选、时间轴、复制等）必须 100% 对齐
2. `papers/<会议>/<年份>.txt` 这套源数据格式不变
3. 部署到 GitHub Pages，URL 不变

### 1.4 顺便要做的优化

1. UI 重新设计，更现代更干净
2. 删掉 polyfill.io（安全风险）
3. Font Awesome 换 lucide-react（按需引入图标）
4. 加 `.gitignore` 排除生成的数据文件

## 3. Git 分支基础

> 这一章必须看懂，否则后面会迷路。

### 3.1 分支是什么

把你的项目想象成一本书。git 帮你管理这本书的修订历史。

**主分支（main）** 就是这本书的"正式发行版"。

**新建一个分支（branch）** 相当于复印一份正式版本，你在复印件上随便涂改、写新章节。**不管你在复印件上做什么，正式版本一行字都不会变**。

**切换分支（checkout）** 就是把"桌面上摆着的那本书"换一本。你切到 `v2-react` 分支，桌上的书就是复印件；你切回 `main`，桌上的书又变回正式版。

**关键认知**：
- 文件**不会消失**，只是"显示哪个版本"在变
- 在新分支上改的东西，**不会影响**原分支
- 可以来回切换，**随时**切回 main 看原始版本
- 切换前要 `git status` 看一下有没有没保存的修改，免得丢东西

### 3.2 本项目的分支布局

```
main              ← 老的 vanilla 版本，保持不动，GH Pages 暂时还从这里发布
v2-react          ← 新的 React+TS 重构，你的工作分支
gh-pages          ← 自动构建产物，GH Pages 最终从这里发布（CI 自动维护）
```

### 3.3 常用命令速查

```bash
# 看现在在哪个分支
git branch

# 从 main 创建并切到 v2-react 分支（一次性命令）
git checkout main
git checkout -b v2-react

# 在 v2-react 分支上正常工作
git add .
git commit -m "init: vite + react + ts 脚手架"
git push -u origin v2-react   # 第一次推送

# 想切回 main 看一眼老代码？
git checkout main
# ... 看完 ...
git checkout v2-react         # 切回去继续

# 想把 main 上的新提交合到 v2-react？
git checkout v2-react
git merge main
```

### 3.4 万一搞砸了

```bash
# 看最近几次提交
git log --oneline -10

# 想回到某次提交的状态（保留改动到工作目录）
git reset --soft <commit-hash>

# 完全丢弃当前所有改动，回到上次 commit
git reset --hard HEAD

# 不小心 commit 到错的分支了
git reset --soft HEAD~1       # 撤销最后一次 commit，改动保留
git stash                     # 暂存改动
git checkout <对的分支>
git stash pop                 # 恢复改动
```

### 4.2 各文件夹职责

- **`papers/`** 源 txt 数据，只增不删，必须进 git。
- **`scripts/`** 任何"开发期跑的"脚本（不是网页代码）。
- **`src/components/ui/`** shadcn 复制过来的"原子组件"，不要随便改。
- **`src/components/features/`** 业务组件（搜索、表格、时间轴），可以随便改。
- **`src/hooks/`** 复用的状态逻辑，前缀必须是 `use`。
- **`src/lib/`** 纯函数工具，不依赖 React，可以单测。
- **`src/types/`** 类型定义集中地。
- **`public/`** 直接被 Vite 拷贝到产物目录，URL 路径就是文件路径（如 `public/data/manifest.json` → `/data/manifest.json`）。
- **`public/data/`** 数据脚本生成，**不进 git**。

### Phase 3: 数据加载层

**目标**：写好 `useManifest` 和 `usePapers` hooks，实现"按需加载"。

**步骤**：

1. `src/hooks/useManifest.ts`：
   ```ts
   import { useEffect, useState } from 'react';
   import type { Manifest } from '@/types';

   export function useManifest() {
     const [manifest, setManifest] = useState<Manifest | null>(null);
     const [error, setError] = useState<Error | null>(null);

     useEffect(() => {
       fetch(`${import.meta.env.BASE_URL}data/manifest.json`)
         .then(r => r.json())
         .then(setManifest)
         .catch(setError);
     }, []);

     return { manifest, error };
   }
   ```

2. `src/hooks/usePapers.ts`：
   ```ts
   // 接收一组 (conference, year) 对，按需 fetch，合并返回 Paper[]
   ```

## 6. UI 设计方向

### 6.1 视觉语言

**字体**：Inter（英文）+ 系统中文（用户系统的中文字体即可，不强制）

```css
font-family: 'Inter', 'PingFang SC', 'Microsoft YaHei', sans-serif;
```

**色板**（基于 Tailwind zinc + indigo）：

| 用途 | 亮色 | 暗色 |
|---|---|---|
| 背景 | `bg-zinc-50` | `bg-zinc-950` |
| 卡片背景 | `bg-white` | `bg-zinc-900` |
| 边框 | `border-zinc-200` | `border-zinc-800` |
| 主文字 | `text-zinc-900` | `text-zinc-100` |
| 次文字 | `text-zinc-500` | `text-zinc-400` |
| 主色（强调） | `bg-indigo-600 text-white` | `bg-indigo-500 text-white` |
| 主色 hover | `bg-indigo-700` | `bg-indigo-600` |

**会议 badge 配色**（用 Tailwind 的颜色家族区分领域）：
- CV → `bg-blue-100 text-blue-700` / 暗 `bg-blue-950 text-blue-300`
- AI → `bg-amber-100 text-amber-700` / 暗 `bg-amber-950 text-amber-300`
- ML → `bg-emerald-100 text-emerald-700` / 暗 `bg-emerald-950 text-emerald-300`

**圆角**：统一用 `rounded-md`（6px），按钮和卡片都这样，不混用。

**间距**：尽量用 `4`、`6`、`8`、`12` 这几档 Tailwind 间距，避免 `3`、`5`、`7` 这种奇数。

**阴影**：尽量不用 `shadow-lg` 这种重的，hover 时给个 `shadow-sm` 就够。

### 6.2 布局

**桌面端**（≥ 1024px）：
```
┌─────────────────────────────────────────┐
│ Header（搜索 + 筛选，sticky）              │
├──────────┬──────────────────────────────┤
│          │                              │
│ 时间轴   │  论文表格                     │
│          │                              │
│ (侧栏)   │                              │
│          │                              │
│          │  分页                         │
└──────────┴──────────────────────────────┘
```

**移动端**（< 1024px）：
```
┌────────────────┐
│ Header         │
├────────────────┤
│ [时间轴折叠]    │  ← Accordion 默认收起
├────────────────┤
│ 论文表格        │
└────────────────┘
```

### 6.3 微动画

- 按钮 hover：`transition-colors duration-150`
- 表格行 hover：`hover:bg-zinc-50 dark:hover:bg-zinc-900/50`
- 复制成功：`<Toast>` 从右下角滑出
- Loading：用 shadcn `<Skeleton>` 占位

### 6.4 暗色模式细节

- 用 Tailwind `dark:` 前缀
- 切换时整个页面有 200ms 的 `transition-colors`
- 默认跟随系统（`window.matchMedia('(prefers-color-scheme: dark)')`），用户手动切过之后用 localStorage 记住

## 7. 关键决策记录

**为什么不一次性把所有 conferences 加载？**
首屏速度。默认只加载选中的（会议, 年份）组合。

**为什么 data 不进 git？**
仓库会膨胀，且数据可以从 papers/*.txt 100% 还原。

**为什么用 hash 而不是时间戳做版本号？**
hash 只在内容变化时变，浏览器能正确缓存。时间戳每次都变。

**为什么不用 TanStack Query？**
项目规模小，自己写一个 50 行的 hook 比引入 30KB 的库划算。

**为什么是 v2-react 分支而不是直接在 main？**
保护现有用户。重构期间 main 上的老版本继续服务。

**为什么用 shadcn 而不是 MUI / Ant Design？**
shadcn 是"复制粘贴源代码"形式，没有 npm 体积负担。MUI / Ant 整套引入会增加 100KB+。

**为什么不用 Preact？**
shadcn / Radix 等库对 React 兼容更好。Preact 省的 35KB 在数据量面前不是瓶颈。

## 8. 坑点速查

### 8.1 Vite base path

GH Pages 路径是 `username.github.io/<repo>/`，必须设：
```ts
// vite.config.ts
base: '/<repo>/'
```
否则所有 `<script>`、`<link>`、`fetch` 都会 404。

**fetch 数据时**：用 `import.meta.env.BASE_URL` 拼路径，不要硬编码 `/data/...`：
```ts
fetch(`${import.meta.env.BASE_URL}data/manifest.json`)
```

### 8.2 React StrictMode 双调用

开发模式下 `useEffect` 会跑两次（这是故意的，帮你发现副作用问题）。看到 fetch 请求发了两次别慌，生产构建是正常的一次。

### 8.3 Tailwind 类名要写全

不能动态拼接 className 字符串：
```tsx
// 错：Tailwind 编译时扫不到，样式不会生效
<div className={`bg-${color}-500`}>

// 对：写完整类名
<div className={color === 'red' ? 'bg-red-500' : 'bg-blue-500'}>
```

### 8.4 MathJax 时机

MathJax 通过 `<script async>` 加载，渲染时要等它 ready。`better-react-mathjax` 帮你处理了。如果用全局 MathJax：
```ts
if (window.MathJax?.typesetPromise) {
  window.MathJax.typesetPromise([element]);
}
```

### 8.5 LocalStorage 在 SSR / 首次渲染

读 localStorage 必须在 `useEffect` 里，不能在组件顶部：
```ts
// 错：服务端渲染会崩
const [dark, setDark] = useState(localStorage.getItem('dark') === 'true');

// 对：
const [dark, setDark] = useState(false);
useEffect(() => {
  setDark(localStorage.getItem('dark') === 'true');
}, []);
```
（不过你是纯 CSR，不太会踩到，但养成习惯）

### 8.6 大列表性能

如果论文表格一页超过 100 行，考虑虚拟滚动（`react-window` 或 `@tanstack/react-virtual`）。一页 50 条就不用了。

### 8.7 gh-pages 分支历史膨胀

用 `force_orphan: true` 让每次部署只保留一次提交，否则分支会越来越大。

### 8.8 fetch URL 里的 hash 不要写错

```ts
// manifest 里：{ file: 'data/nips/2024.json', hash: 'abc' }
fetch(`${import.meta.env.BASE_URL}${meta.file}?v=${meta.hash}`)
// → /<repo>/data/nips/2024.json?v=abc ✓
```

### 8.9 papers 目录路径

`gen-data.ts` 里的路径要相对于运行命令的 CWD，不是脚本文件位置。建议用 `path.resolve(process.cwd(), 'papers')` 显式定位。

## 10. 备忘录

### 10.1 老版本如何应急维护

如果 v2 开发期间 main 上老版本有 bug 要修：

```bash
git checkout main
# ... 改 docs/ 下的文件 ...
python3 pre_data.py     # 如果改了 papers
git add . && git commit -m "fix: ..."
git push

git checkout v2-react   # 切回工作分支继续
```

### 10.2 老版本下线节奏

- v2 上线后老 `docs/` 可保留 1-2 周作为应急回滚备份
- 没问题后 `git rm -r docs` 删除
- 老版本永远在 git 历史里，需要时 `git checkout <旧提交>` 还能看

### 10.3 加新会议怎么办

1. 在 `papers/` 下新建会议目录，加 `.txt` 文件
2. 在 `scripts/gen-data.ts` 的 `CONFERENCE_META` 加上新会议的元数据
3. 本地 `npm run gen:data` 验证
4. `git add papers/ scripts/ && git commit && git push`
5. CI 自动部署

### 10.4 添加新功能的标准流程

1. 在 v2-react（或 main，看时机）分支上
2. 写代码 → 本地 `npm run dev` 验证
3. `npm run build` 确认能编译
4. commit + push
5. CI 自动构建部署
